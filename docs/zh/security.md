## 报告漏洞

我们非常重视安全漏洞。如果您在 MomShell 中发现安全问题，请负责任地报告。

### 报告方式

**请勿通过公开的 GitHub Issues 报告安全漏洞。**

请通过以下方式之一报告：

1. **GitHub 安全公告（推荐）**
   [报告漏洞](https://github.com/koishi510/MomShell/security/advisories/new)

2. **电子邮件**
   直接联系维护者（见仓库维护者个人资料）

### 报告内容

请在报告中包含以下信息：

- 漏洞类型（如 SQL 注入、XSS、认证绕过）
- 受影响的组件（心灵伙伴、姐妹纽带、回声/回忆录、照片画廊、悄悄话、任务、管理面板、API 等）
- 复现问题的分步说明
- 概念验证或利用代码（如可能）
- 漏洞的潜在影响
- 建议的修复方案（可选）

### 处理流程

- **确认收到**：我们将在 48 小时内确认收到您的报告
- **评估**：我们将调查并评估问题的严重程度
- **更新**：我们将持续通知您进展情况
- **解决**：我们会尽快解决关键漏洞
- **致谢**：我们将在发布说明中致谢（除非您希望匿名）

### 部署安全最佳实践

部署 MomShell 时，请确保：

1. **环境变量**
   - 永远不要将 `.env` 文件或 API 密钥提交到版本控制
   - 使用安全的密钥管理方案管理 `OPENAI_API_KEY`
   - 在生产环境中将 `JWT_SECRET_KEY` 更改为安全的随机值
   - 定期轮换 API 密钥

2. **网络安全**
   - 在生产环境中使用 HTTPS（在 Nginx 中配置 SSL/TLS）
   - 限制 API 访问来源（CORS 配置）
   - 使用防火墙限制暴露的端口

3. **数据库安全**
   - 使用强唯一密码访问 PostgreSQL
   - 定期备份数据库
   - 限制数据库端口的网络访问

4. **Docker 安全**
   - 保持基础镜像更新
   - 生产环境中不要以 root 运行容器
   - 使用 Docker secrets 管理敏感数据

5. **内容审核**
   - 根据部署需求审查和自定义敏感内容过滤器
   - 定期监控被标记的内容
   - 确保危机干预流程正确配置

## 范围

以下属于安全报告范围：

- 认证和授权漏洞
- 数据暴露或泄露
- 注入攻击（SQL、命令等）
- 跨站脚本攻击（XSS）
- 跨站请求伪造（CSRF）
- 不安全的直接对象引用
- 默认设置中的安全配置错误
- 依赖项中的漏洞

以下**不在范围内**：

- 拒绝服务（DoS）攻击
- 社会工程攻击
- 物理安全问题
- 第三方服务的问题（ModelScope API 等）
- 需要物理访问服务器的问题
